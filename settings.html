Engines which engine runs the work

Chooses where splitting and lyric transcription happen. Leave both on Auto and it does the right thing: use the server below if it's running, otherwise fall back to the in-app engine if you installed one.

“Server” = the managed local demucs server below whenever it's running, otherwise whatever the app's demucs_server_url points at — so the same option covers both. “In-app” = the models run inside the feedBack process itself.
These settings apply to both: “Split model” is sent to the server and used by the in-app audio-separator; “WhisperX model” is the one exception — only the in-app transcriber uses it, the server has its own.

Local demucs server recommended — start here

The easiest way to get splitting working. Runs got-feedBack's real demucs/whisperx server on this machine and manages it for you. Install server + models does the whole setup in one click (dependencies, then the model weights) and starts it.

Once it's running, the plugin uses it automatically — nothing else to configure. It can use your GPU (big speed-up: seconds instead of minutes), it runs in its own process so it can't destabilise the app, and Uninstall server removes every byte it downloaded. This is the only thing here that downloads anything; app startup never downloads and is never blocked by it.

Local engine alternative — no server

  
Runs the models inside the feedBack process instead of talking to
    a server. You don't need this if you're using the server above — it's a separate, second download.

  
Worth knowing: it's CPU-only (no CUDA install path, so it's slower
    than the server on an NVIDIA machine), it loads multi-GB models into the app's own process, and its
    weights land in the shared ~/.cache — so Uninstall can't reclaim them all.
    Prefer the server unless you specifically want to avoid running one.
